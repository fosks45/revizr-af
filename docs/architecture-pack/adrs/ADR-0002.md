# ADR-0002: Backend Language and Framework (TypeScript / Node.js + Fastify)

**Status:** Accepted
**Date:** 2026-05-15
**Feature:** 002-revizr
**Author:** solution-architect

---

## Context

Revizr requires a backend API serving: authentication, diagnostic job submission and
status polling (async, up to 60 s), question retrieval and practice session management,
parent dashboard data, subscription webhooks from Stripe, GDPR rights endpoints, and
admin content management. The API must also proxy requests to the Claude AI provider
(diagnostic engine) — no browser code may call the AI API directly (Principle 10 and
compliance C-013 require the gateway pattern).

The backend must support Server-Sent Events (SSE) for diagnostic status streaming and
must coexist cleanly with the Next.js frontend in a monorepo or a separate service
deployment. The performance target is p95 API response ≤ 800 ms for question load on
the low-end tier.

Language choice is constrained by the monorepo preference: a single language across
frontend and backend reduces context-switching cost and enables shared types for the
API contract (Zod schemas shared between Next.js route handlers and the backend).

---

## Decision

**Node.js 22 LTS with TypeScript 5, using Fastify 5 as the HTTP framework.**

Fastify is selected over Express for the following specific reasons:
1. **Performance:** Fastify's JSON serialisation (via `fast-json-stringify` with schemas)
   and routing (via `find-my-way` Radix tree) benchmarks at 3–4× Express throughput
   for JSON API workloads. The 800 ms p95 SLO requires minimal framework overhead.
2. **Schema-first:** Fastify's native integration with JSON Schema for request validation
   and response serialisation enforces the OpenAPI contract at the framework layer —
   a request that does not match the schema is rejected before it reaches the handler.
   This is the backend enforcement of the OpenAPI contract defined in `contracts/openapi.yaml`.
3. **SSE support:** Fastify supports streaming responses natively via `reply.raw` or
   the `@fastify/sse` plugin. The diagnostic polling endpoint emits SSE events;
   Fastify handles this without requiring a custom HTTP server.
4. **Plugin ecosystem:** `@fastify/auth`, `@fastify/jwt`, `@fastify/rate-limit`,
   `@fastify/multipart` (for document upload), `@fastify/swagger` (OpenAPI doc gen)
   are well-maintained official plugins.
5. **TypeScript first-class:** Fastify 5 ships with TypeScript type definitions; typed
   route schemas with Zod via `fastify-type-provider-zod` provide end-to-end type
   safety from HTTP layer to database query.

The AI gateway (Claude API proxy) is implemented as a dedicated Fastify plugin, not a
standalone service, for v1. The plugin intercepts all outbound Claude API calls, logs
`prompt_hash`, `response_hash`, `model_id`, `agent_version`, `tokens_in`, `tokens_out`,
`latency_ms`, and `cost_usd` to the audit ledger (Principle 10). It also enforces the
data processing constraint: no raw school report content is ever sent to the Claude API
without the AI provider DPA being confirmed in place (Condition C-013).

**Monorepo structure:** `apps/api` (Fastify server) and `apps/web` (Next.js) share
`packages/types` (Zod schemas + inferred TypeScript types) and `packages/db`
(Drizzle ORM client + migrations). Turborepo manages the monorepo build graph.

---

## Consequences

**Positive:**
- Shared TypeScript types between frontend and backend eliminate entire classes of
  contract drift bugs — the API response type is exactly what the React component
  expects, enforced at compile time.
- Fastify's schema validation means malformed requests are rejected with a consistent
  422 response without custom middleware code.
- Node.js's event loop handles SSE connections efficiently; the diagnostic polling
  endpoint maintains a long-lived SSE connection per active diagnostic job without
  blocking a thread.
- The AI gateway plugin centralises all model call logging, enforcing Principle 10
  across all diagnostic invocations without requiring individual handlers to implement
  logging.

**Negative / Trade-offs:**
- Node.js is single-threaded; CPU-intensive work (image processing, bulk question
  imports) must be offshored to worker threads or a queue to avoid blocking the event
  loop. Mitigated by using BullMQ (Redis-backed) job queue for diagnostic processing
  — the Fastify handler submits the job and returns a job ID immediately; the worker
  pool processes the job asynchronously.
- Fastify is less ubiquitous than Express in tutorials and Stack Overflow examples.
  Mitigated by official Fastify documentation being comprehensive and the team being
  TypeScript-native.

---

## Alternatives Considered

| Alternative | Reason Rejected |
|---|---|
| **Express.js** | Express is not type-safe by default; adding TypeScript types is bolted-on, not intrinsic. Express has no native JSON schema validation — request validation requires additional libraries. Express is 3–4× slower than Fastify on JSON API benchmarks. Rejected on performance and type-safety grounds. |
| **Go (net/http or Gin)** | Go would give excellent performance and low memory footprint. Rejected because it splits the language stack: frontend is TypeScript, backend would be Go, eliminating shared types. Hiring full-stack TypeScript engineers is significantly easier than hiring Go engineers in the UK startup market. |
| **Python (FastAPI)** | FastAPI has strong type safety via Pydantic. Rejected because: (a) language split with TypeScript frontend; (b) Python async performance is adequate but not superior to Node.js for I/O-heavy API workloads; (c) same hiring argument as Go. |
| **Deno / Bun** | Both are interesting runtimes but are not LTS-stable with the same production track record as Node.js 22 LTS. Rejected on maturity grounds for a compliance-sensitive product. |
| **Hono (edge-first)** | Hono is an excellent edge-compatible framework. Rejected because the compliance constraint (C-012) requires UK-only data residency, and edge-first deployment models assume globally distributed execution. Hono is better suited for a product without data residency constraints. |
