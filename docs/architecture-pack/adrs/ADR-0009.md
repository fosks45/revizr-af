# ADR-0009: UK Data Residency Enforcement

**Status:** Accepted
**Date:** 2026-05-15
**Feature:** 002-revizr
**Author:** solution-architect

---

## Context

Revizr processes data for UK children (C7), UK school performance data (C5 potential),
and UK customer PII (C3). The compliance pack establishes:

- **C-012**: No personal data (C3 or above) may be held or processed outside the
  United Kingdom. UK GDPR (retained EU law, as amended by the Data Protection Act
  2018 and the Data Protection (Adequacy) (United States of America) Order 2025)
  applies. Transfers to countries without UK adequacy decisions require Article 46
  transfer mechanisms.
- **GDPR (UK) Article 46**: International data transfers require appropriate
  safeguards (UK adequacy decisions, binding corporate rules, or standard contractual
  clauses). The simplest compliance path is to keep all personal data in the UK and
  avoid transfers entirely.
- **C7 controls**: Children's data under C7 triggers "strictest of all" cross-border
  rules — no cross-border transfer, sovereign residency. Revizr's primary persona
  (Amara, age 15) is within scope.

The architecture spans multiple vendors (AWS, Cloudflare, Vercel, Anthropic/Claude,
Stripe, Resend). Each vendor must be assessed for data residency impact.

The question: how is UK residency enforced mechanically, and which vendors are
in scope for personal data processing?

---

## Decision

**All C3+ data stores are restricted to AWS eu-west-2 (London). Cloudflare is used
for static assets only — no personal data passes through CDN edge nodes. Vercel
deployment is pinned to London. Claude API is called with extracted text only (no
raw documents containing C3+ identifiers are sent). All vendor DPAs are in place
before production launch.**

Enforcement per layer:

**1. Database (PostgreSQL via RDS or self-managed on EC2/ECS):**
- Hosted in AWS eu-west-2 exclusively.
- No read replicas outside eu-west-2.
- Database connection pool in Fastify is configured with a hardcoded endpoint in
  eu-west-2; no regional failover to non-UK regions.
- RDS parameter group and subnet group scoped to eu-west-2 availability zones only.

**2. Object storage (S3):**
- Bucket in eu-west-2 with explicit bucket policy `Deny` on any `PutBucketReplication`
  or cross-region replication action.
- S3 Block Public Access enabled.
- SSE-KMS with CMK in KMS eu-west-2.

**3. Application servers (Fastify API + Next.js frontend):**
- Deployed as Docker containers on AWS ECS Fargate in eu-west-2.
- Alternatively: Vercel Pro with deployment region pinned to `lhr1` (London Heathrow
  PoP). Vercel's edge runtime is NOT used for any route that reads, writes, or
  processes personal data — only static and ISR routes may use the edge runtime.
- Environment variables confirm the AWS region at startup; the application refuses
  to start if `AWS_DEFAULT_REGION` is not `eu-west-2`.

**4. CDN (Cloudflare):**
- Cloudflare is used exclusively for static asset delivery (JS bundles, CSS, fonts,
  images) at the Cloudflare Free/Pro tier.
- All Cloudflare cache rules are configured with `Cache-Control: no-store` for any
  route that could contain personalised content (i.e., any route behind `/api/` or
  authenticated routes).
- No personal data is ever present in a Cloudflare-cached response. The Cache-Control
  header on all API responses is `no-store, no-cache, private`.
- Cloudflare's Zaraz and other analytics features are disabled; no user behavioural
  data flows through Cloudflare edge nodes.

**5. AI API (Claude / Anthropic):**
- The Claude API is a US-based service. Under ADR-0004, all calls are routed through
  the Revizr server-side gateway.
- The gateway enforces a scrubbing step before any data leaves the UK: student
  names, school names, and direct identifiers are replaced with tokens before the
  prompt is assembled. Only anonymised or pseudonymised text is sent to Anthropic.
- Anthropic's Data Processing Agreement (DPA) must be in place under UK GDPR
  Article 28 before any real user data is processed. This is a blocking human-
  approval item in the Phase 7 gate.
- Because the data sent to Anthropic is pseudonymised (tokens replace identifiers),
  it is arguable this does not constitute a transfer of personal data. Legal review
  is required to confirm this position; pending legal review, the DPA is mandatory.

**6. Email (Resend):**
- Resend processes transactional email. Email addresses (C3) are transmitted to
  Resend for delivery.
- Resend must have a UK GDPR-compliant DPA in place. Resend is EU/EEA-hosted (Dublin);
  under the UK-EU adequacy decision, this is a permitted transfer.
- No email content contains sensitive personal data (no diagnostic results, no
  child data) — subject lines and bodies are generic notification templates.

**7. Payment (Stripe):**
- Stripe is a US entity. Card data never touches Revizr's systems (ADR-0007).
- The only data sent to Stripe is the user's email address and the Stripe customer
  ID. Stripe's DPA (standard, available in the Stripe Dashboard) covers this under
  UK GDPR Article 46 standard contractual clauses.

**8. Redis / BullMQ:**
- Redis instance (ElastiCache or self-managed) is deployed in eu-west-2.
- Job payloads in BullMQ queues contain only job IDs, not raw personal data.
  The worker fetches data from PostgreSQL using the job ID.

---

## Consequences

**Positive:**
- UK residency is enforced at the infrastructure layer, not just policy. Mechanical
  controls (bucket policy deny, hardcoded region checks, Cloudflare cache rules)
  cannot be bypassed by application code without explicit configuration changes.
- Keeping all data in one region simplifies the data inventory and ROPA (Record of
  Processing Activities) required under UK GDPR Article 30.
- No cross-border transfer mechanism (SCCs, BCRs) is needed for the primary data
  stores, removing a significant compliance maintenance burden.

**Negative / Trade-offs:**
- Single-region architecture means a full AWS eu-west-2 outage would take down
  Revizr entirely. Mitigated by: the educational use case tolerates outages better
  than financial services; RTO/RPO targets (12h/1h) are achievable with backups
  in a separate AWS account in eu-west-2.
- The Claude API PII-scrubbing step adds latency (~20–50 ms) to every diagnostic
  request. This is within the acceptable range for an async job that already takes
  5–30 seconds.
- Legal review of the Claude API pseudonymisation position may conclude that DPA
  alone is insufficient and UK Article 46 SCCs are required. The DPA is being treated
  as the minimum; SCCs may be added. This is a human-approval item.
- Vercel edge runtime restriction means some performance optimisations (edge SSR,
  edge middleware for auth) are unavailable for authenticated routes. Mitigated by
  deploying API routes on ECS Fargate with fast cold-start times.

## Alternatives Considered

| Alternative | Reason Rejected |
|---|---|
| **AWS eu-west-1 (Ireland) as fallback region** | Ireland is EEA, covered by UK-EU adequacy decision. Technically permissible but C-012 says "United Kingdom." Using eu-west-1 as a warm standby would require legal confirmation that the adequacy decision covers all data classes including C7. Deferred to v2 disaster recovery planning. |
| **Use Cloudflare for API caching (personal data at edge)** | Cloudflare's global network has PoPs outside the UK and EU. Caching authenticated API responses at edge would violate C-012. Rejected. |
| **Azure UK South as primary region** | Equivalent UK residency, but Revizr's infrastructure is AWS-first. Mixed cloud in v1 adds operational overhead for no benefit. |
| **Vercel data residency via Enterprise plan** | Vercel Enterprise offers region-locked deployments with stronger residency guarantees. The Pro plan with `lhr1` pinning is sufficient for v1; upgrade to Enterprise is a trigger if data residency audit requires stronger contractual guarantees from Vercel. |
