# ADR-0006: Async Job Pattern for Diagnostic Engine (Job Queue + SSE)

**Status:** Accepted
**Date:** 2026-05-15
**Feature:** 002-revizr
**Author:** solution-architect

---

## Context

The diagnostic engine (F2: report upload; F3: in-product quiz) involves calling the
Claude AI API to parse a school report or analyse quiz responses and produce a topic
weakness map. The performance budget for this operation is p95 ≤ 60 s (low-end tier),
≤ 45 s (standard tier), with a hard spec.md NFR of ≤ 90 s.

A synchronous HTTP request for this operation is not viable because:
1. HTTP request timeouts at load balancers and proxies typically default to 30–60 s.
   A p95 of 60 s means some requests will take 80–90 s, which exceeds common timeout
   limits.
2. Synchronous processing blocks the Fastify event loop for the duration of the AI
   API call if not handled carefully.
3. If the AI API is slow or rate-limited, a synchronous approach provides no mechanism
   to queue and retry the request without holding the user's browser connection open
   for an indefinite period.
4. The design pack specifies a dedicated processing screen (OB-08) with a spinner and
   "We're reading your report. This takes up to 90 seconds." — this is a first-class
   UX state that requires the browser to receive status updates during processing.

The solution must also operate correctly on low-end devices (Amara's tablet) where the
browser tab may be backgrounded during the 60-second processing window.

---

## Decision

**BullMQ (Redis-backed) job queue for diagnostic processing, with Server-Sent Events
(SSE) for real-time status delivery to the browser.**

Architecture:

```
Browser                    Fastify API              BullMQ Worker          Claude API
  |                             |                        |                      |
  |--- POST /diagnostics ------>|                        |                      |
  |                             |-- enqueue job -------->|                      |
  |<-- 202 { jobId } -----------|                        |                      |
  |                             |                        |                      |
  |--- GET /diagnostics/:id/status (SSE) -->|            |                      |
  |                             |                        |                      |
  |                             |              |-- process job --------------->|
  |                             |              |<-- AI response ---------------|
  |                             |              |-- persist weakness map ------->|
  |                             |              |-- emit job:complete ---------->|
  |<-- SSE: {status: 'processing', progress: 30} --|     |                      |
  |<-- SSE: {status: 'complete', weaknessMapId}  --|     |                      |
  |                             |                        |                      |
```

**Components:**

1. **POST `/api/v1/diagnostics`** — accepts the job request (document reference or quiz
   results), enqueues a BullMQ job, returns `202 Accepted` with `{ jobId, statusUrl }`.
   The upload is stored to S3 (encrypted) during this request; the S3 object key is the
   job payload — not the document content itself (which never touches the API layer directly).

2. **BullMQ Worker** (separate Node.js process, same monorepo, `apps/worker`) — picks up
   the job, retrieves the document from S3, calls the Claude API gateway, validates the
   structured output, persists the weakness map to PostgreSQL, deletes the S3 object
   (data minimisation: BR-F2-01), updates job status to `complete`. Retry budget: 2
   retries with exponential backoff (initial delay 5 s). After 2 failures: job marked
   `failed`, user notified via SSE.

3. **GET `/api/v1/diagnostics/:jobId/status`** (SSE endpoint) — the browser connects
   immediately after receiving the `202`. The SSE endpoint subscribes to BullMQ job
   events via the BullMQ `QueueEvents` class and streams status updates to the browser.
   Events emitted: `queued`, `processing` (with estimated percentage), `complete` (with
   weaknessMapId for redirect), `failed` (with fallback action URL).

4. **Polling fallback** — if SSE is not supported by the browser or if the SSE connection
   drops (network interruption), the frontend falls back to polling
   `GET /api/v1/diagnostics/:jobId/status?poll=true` every 5 seconds. The polling
   endpoint returns the current job status as JSON. This handles the case where Amara's
   tablet backgrounds the tab — on return, the frontend polls for the current status and
   recovers gracefully.

5. **Redis** — BullMQ requires a Redis instance. Hosted on AWS ElastiCache (Redis 7)
   in eu-west-2. Redis stores job state only (metadata, status, retry count) — no
   personal data is stored in Redis. Job payloads in Redis contain only the S3 object
   key and job parameters, not document content.

**Progress estimation:** The worker emits progress events during the AI API call by
tracking the streaming response from Claude. As tokens are received, the estimated
percentage complete is updated based on the expected response length. This allows the
OB-08 processing screen to show meaningful progress rather than a static spinner.

---

## Consequences

**Positive:**
- The 202/SSE pattern decouples the browser connection lifetime from the AI API call
  duration — the browser can reconnect at any time and pick up the current job status.
- BullMQ's retry logic handles transient AI API failures (rate limits, 5xx errors)
  without user intervention.
- The worker process can be scaled horizontally (multiple worker instances) to handle
  peak diagnostic volume without affecting API response times.
- Document deletion from S3 happens in the worker immediately after processing —
  the shortest possible retention window for C5 special-category data (BR-F2-01).

**Negative / Trade-offs:**
- Redis is an additional infrastructure component. At v1 scale, a single
  ElastiCache t3.micro instance is sufficient (~£20/month in eu-west-2). This
  is a justified cost given the job queue requirement.
- BullMQ workers are a separate deployment unit from the Fastify API. Monorepo
  management via Turborepo handles this cleanly; CI must include worker deployment.
- SSE requires the browser to maintain an open HTTP/2 connection. This is not an
  issue on modern browsers; older browsers (IE11 — not in the support matrix per
  platform-decision) would require the polling fallback.

---

## Alternatives Considered

| Alternative | Reason Rejected |
|---|---|
| **WebSockets** | WebSockets provide bidirectional communication, but diagnostic status is unidirectional (server → browser). WebSockets add complexity (connection management, message framing) for a use case that SSE handles natively. SSE is a simpler, one-way streaming protocol sufficient for this job. Rejected in favour of SSE. |
| **Long-polling (HTTP)** | Long-polling holds an HTTP connection open and returns when an event occurs or on timeout. Works but is less efficient than SSE for sustained updates over 60 seconds — each long-poll response requires a new HTTP connection. Rejected in favour of SSE. |
| **Synchronous API call with extended timeout** | Configure the API gateway and load balancer timeout to 120 s. Simpler implementation. Rejected because: (a) blocking the Fastify event loop or holding threads for 60–90 s degrades concurrent request handling; (b) no progress updates — user sees a spinner for 90 s with no feedback (violates the 200 ms skeleton state requirement from platform-budgets.md); (c) the process provides no retry capability if the AI API times out. |
| **Webhook-based (API calls back when done)** | The backend calls the AI API and receives a webhook when complete. Rejected because: (a) Claude API does not support async webhook callbacks — it requires the caller to hold the connection or stream the response; (b) even if it did, routing a webhook to the correct SSE connection for the specific user's browser session requires additional session mapping infrastructure that BullMQ + SSE avoids. |
