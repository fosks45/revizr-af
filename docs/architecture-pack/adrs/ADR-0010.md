# ADR-0010: PWA Strategy (Service Worker, Install, Offline Deferred)

**Status:** Accepted
**Date:** 2026-05-15
**Feature:** 002-revizr
**Author:** solution-architect

---

## Context

The platform pack (Phase 5) established: Revizr is a PWA from Sprint 0. This means
a service worker and web app manifest are required in v1. The full offline session
feature (F29) is explicitly deferred to v2. The constraints:

1. **Performance budget:** FCP ≤ 2 000 ms on a low-end Chromebook (Amara's device
   tier). Static assets (JS bundles, CSS, fonts, images) must be cache-first to
   avoid repeat-visit network round trips on slow school Wi-Fi.
2. **Install prompt:** The PWA install prompt should appear after the first completed
   study session — not on landing, not immediately on sign-up. Friction-free first
   use; install nudge once value has been demonstrated.
3. **Offline full session (F29) deferred to v2:** In v1, the app requires a network
   connection to function. Users should see a graceful offline page, not a browser
   error. Questions are fetched from the server; answers are submitted to the server.
   No local-first or sync-on-reconnect logic is in scope for v1.
4. **GDPR / data residency:** The service worker caches static assets only. API
   responses containing personal data (C3+) must not be cached by the service worker.
5. **Next.js compatibility:** The service worker strategy must integrate with Next.js
   15 App Router without conflicting with RSC streaming or the `next/font` and
   `next/image` optimisation pipeline.

---

## Decision

**Workbox-based service worker via `@ducanh2912/next-pwa` (the actively maintained
fork of `next-pwa`); cache-first strategy for static assets; network-first for API
routes; install prompt shown after first completed practice session; full offline
session deferred to v2.**

Implementation specifics:

**Service worker:**
- `@ducanh2912/next-pwa` wraps Workbox and integrates with Next.js 15's build output.
  It generates a service worker at build time with precache manifests for the
  `/_next/static/` asset tree and any explicitly configured routes.
- Cache strategy per route type:
  - `/_next/static/**` → `CacheFirst` (immutable assets, content-hashed filenames).
    Max age: 365 days.
  - `/fonts/**`, `/images/**` → `CacheFirst`. Max age: 30 days.
  - `/api/**` → `NetworkFirst` with a 3-second timeout; on network failure, returns
    a cached offline stub response (`{ error: "offline" }`). **No personal data
    responses are cached.** The `Cache-Control: no-store, private` header on all
    API responses prevents Workbox from caching them even if the strategy were
    misconfigured — belt-and-braces.
  - `/` (app shell routes) → `StaleWhileRevalidate` for public routes;
    `NetworkFirst` for authenticated routes.
- The web app manifest (`manifest.json`) declares: `name`, `short_name`, `start_url`,
  `display: standalone`, `background_color`, `theme_color`, `icons` (192px and
  512px maskable). The manifest is served from the Next.js `public/` directory.

**Install prompt:**
- The `beforeinstallprompt` event is captured and suppressed on first fire. The
  prompt is stored in component state.
- After a user completes their first practice session (defined as: `practice_session`
  record with `status = 'completed'`), a non-intrusive banner is shown: "Add Revizr
  to your home screen for quicker access." The stored prompt is triggered on user
  acceptance.
- The banner is dismissed permanently if the user declines (stored in `localStorage`).
- The banner is never shown on desktop Chrome (where the install flow is less
  natural) — detected via the `userAgentData` API or media query heuristic.

**Offline experience (v1 scope):**
- A static offline fallback page (`/offline`) is precached by the service worker.
  If a navigation request fails (no network, no cache), the service worker serves
  the offline page.
- The offline page displays: "You're offline — Revizr needs a connection to load
  your sessions. Your progress is saved. Come back when you're online." No attempt
  to show stale session data in v1.
- All API failures due to network errors surface a `503` stub that the frontend
  handles with a toast notification: "Check your connection and try again."

**Full offline sessions (F29) — deferred to v2:**
- In v2, the plan is: IndexedDB-backed question cache (pre-fetched after session
  start), service worker sync on reconnect, conflict resolution for self-mark
  scores submitted offline. This is out of scope for v1.
- The architecture does not preclude v2: the `NetworkFirst` API strategy can be
  upgraded to `NetworkFirst with offline fallback to IndexedDB` without replacing
  the service worker infrastructure.

---

## Consequences

**Positive:**
- Static asset caching eliminates repeat-visit network overhead for the JS bundles,
  CSS, and fonts on slow school Wi-Fi — directly addresses Amara's device and
  connectivity profile.
- `CacheFirst` for content-hashed assets is safe: Next.js generates new hashes on
  every build, so stale cache never serves outdated code.
- The deferred install prompt strategy reduces bounce rate from premature prompts
  (research: install prompts on landing pages have low acceptance rates). Showing
  after a completed session aligns with demonstrated value.
- Workbox is the industry standard; skills are transferable, documentation is
  thorough, and the v2 offline feature can build on the existing foundation.
- GDPR compliance is maintained: `Cache-Control: no-store, private` on API responses
  is the application-layer guarantee; Workbox `NetworkFirst` for `/api/**` is the
  service-worker-layer guarantee. Two independent controls.

**Negative / Trade-offs:**
- `@ducanh2912/next-pwa` is a community fork, not an official Next.js package.
  It is actively maintained (as of 2026-05) but carries third-party dependency risk.
  Mitigated by pinning to a specific minor version and evaluating the official
  Next.js PWA story at each major Next.js upgrade.
- Service worker updates require a page reload to activate. Users on old tabs see
  the previous version until they refresh. Mitigated by a Workbox `skipWaiting`
  call and a `clients.claim()` in the service worker activate event, paired with
  a silent reload after the service worker updates (only on next navigation, not
  mid-session).
- Full offline deferred to v2 means users without connectivity get a graceful error,
  not a working app. For the school segment, most sessions are on school Wi-Fi or
  home broadband; this is an acceptable v1 trade-off.
- Lighthouse PWA audit in CI (Sprint 7) must pass a score of ≥ 90. Service worker
  registration and manifest validity are gates in the Lighthouse CI configuration.

## Alternatives Considered

| Alternative | Reason Rejected |
|---|---|
| **`next-pwa` (original, by shadowwalker)** | The original `next-pwa` is unmaintained (last commit 2022) and incompatible with Next.js 13+ App Router. Replaced by the `@ducanh2912/next-pwa` fork which has App Router support. |
| **Hand-authored service worker (no Workbox)** | More control, no third-party dependency. However, Workbox handles cache versioning, precache manifest generation, and strategy implementation correctly; reimplementing these is non-trivial and error-prone. The maintenance burden is not justified for v1. Revisit if Workbox becomes a constraint in v2. |
| **Serwist (Next.js-native Workbox integration)** | Serwist is a newer Workbox wrapper with better Next.js App Router integration. It is less battle-tested than `@ducanh2912/next-pwa`. Evaluate at v2 offline feature milestone — may be the correct migration target. |
| **Show install prompt immediately on landing** | Higher install volume in theory, but industry data shows conversion rates for immediate prompts are 5–10× lower than contextual prompts. Rejected on UX grounds. |
| **Full offline in v1** | Would require IndexedDB schema, sync logic, and conflict resolution — estimated 3–4 additional sprint weeks. Does not appear in the 14 Must features. Scope risk to the v1 ship date is unacceptable. Deferred to v2 as F29. |
