# ADR-0007: Payment Processor (Stripe)

**Status:** Accepted
**Date:** 2026-05-15
**Feature:** 002-revizr
**Author:** solution-architect

---

## Context

Revizr requires subscription billing with a free tier and paid monthly/annual plans.
The primary compliance constraints are:

1. **C4 — Customer Financial data** (card numbers, CVV, bank account details) must
   never touch Revizr's servers or be stored in Revizr's database. Any cardholder
   data in scope of PCI DSS dramatically increases audit scope and cost.
2. **GDPR Article 6 lawful basis** for processing financial data requires a Data
   Processing Agreement (DPA) with any payment processor acting as a data processor.
3. **Free-tier gating**: the free tier must allow students to access a capped feature
   set without entering payment details; upgrade must be frictionless and not require
   developer involvement to activate.
4. **Subscription lifecycle events** (trial end, renewal, failure, cancellation)
   must update Revizr's internal subscription state reliably — webhook-driven, not
   client-side callback-driven (client-side is spoofable).
5. **Billing portal self-service**: users must be able to cancel, update payment
   methods, and download invoices without contacting support — mandatory for UK
   consumer-facing SaaS.

---

## Decision

**Stripe Checkout + Stripe Billing Portal, with Stripe Webhooks for all
subscription state transitions. No card data stored in Revizr's database.**

Implementation specifics:

- **Stripe Checkout (hosted):** When a user initiates an upgrade, the Revizr API
  creates a Stripe Checkout Session via the Stripe server-side SDK and returns the
  Checkout URL. The browser redirects to Stripe-hosted pages. Revizr never renders
  a payment form and never touches card data.
- **Stripe Billing Portal (hosted):** For subscription management (cancel, change
  plan, update payment method, download invoices), the API creates a Stripe Billing
  Portal Session. Users are redirected to Stripe's hosted portal, then back to
  Revizr on completion.
- **Webhook-driven state:** All subscription state changes (created, updated,
  trial_will_end, payment_failed, cancelled) are received via Stripe webhooks on
  `POST /subscriptions/webhook`. This endpoint verifies the Stripe-Signature header
  before acting. Revizr's `subscriptions` table is the source of truth for
  entitlement checks — seeded and maintained exclusively from webhook events.
- **Data stored in Revizr:** Only `stripe_subscription_id` (C3) and
  `stripe_customer_id` (C3) are stored — opaque identifiers, not card data. No
  PAN, no CVV, no expiry, no bank details ever enter Revizr's data stores.
- **Free tier:** Implemented entirely within Revizr — no Stripe object created until
  a user initiates upgrade. Feature gates check the `subscriptions.plan` field;
  `free` is the default for all new users.
- **Stripe DPA:** Stripe's standard Data Processing Agreement covers GDPR Article
  28 requirements. Stripe is certified PCI DSS Level 1. Revizr operates as a
  merchant under SAQ A (the lowest-scope PCI questionnaire, available to merchants
  who fully outsource card processing).

---

## Consequences

**Positive:**
- PCI DSS scope is SAQ A — the simplest self-assessment, no annual on-site audit
  required at this stage. Card data never enters Revizr's network or storage.
- Stripe's hosted Checkout and Portal eliminate the need to build and maintain
  payment UI, reducing attack surface and development time.
- Subscription lifecycle is reliable: Stripe retries failed payments automatically,
  sends dunning emails, and emits webhook events for every state transition.
- Stripe's SDK is well-maintained with strong TypeScript types; integration with
  Fastify is straightforward.
- Stripe Billing Portal satisfies the UK Consumer Duty requirement for easy
  cancellation without contacting support.

**Negative / Trade-offs:**
- Stripe takes a percentage fee on each transaction (currently 1.4% + 20p for
  European cards). This is the cost of outsourcing PCI compliance.
- Webhook delivery is eventually consistent — Revizr must handle idempotent
  re-delivery and out-of-order events (e.g., `customer.subscription.updated` arriving
  before `checkout.session.completed` in rare cases). Mitigated by idempotency keys
  and upsert logic in the webhook handler.
- Stripe Checkout redirects away from the Revizr UI, which may interrupt flow.
  Mitigated by a designed success/cancel return URL with a post-upgrade confirmation
  page in Revizr.
- Commercial agreement with Stripe must be confirmed before real transactions are
  processed (flagged as a human-approval item in the Phase 7 gate record).

## Alternatives Considered

| Alternative | Reason Rejected |
|---|---|
| **Paddle** | Paddle acts as Merchant of Record, handling VAT and sales tax globally. Strong fit for future international expansion, but adds complexity (pricing via Paddle dashboard, less developer-friendly webhooks). Revizr is UK-only in v1; Stripe's simplicity wins for now. Revisit at internationalisation milestone. |
| **GoCardless (Direct Debit)** | Suited for recurring subscriptions via Direct Debit, lower fees for high-volume. Requires UK bank account setup per user; much higher friction for students and parents than card-on-file. Rejected on UX grounds for v1. |
| **Braintree (PayPal)** | Comparable feature set to Stripe but more complex SDK, weaker TypeScript types, and less alignment with UK startup hiring knowledge. No material advantage over Stripe for this use case. |
| **Custom payment form + Stripe Elements** | Would keep users on Revizr's domain but brings card data into Revizr's JavaScript context, raising PCI scope to SAQ A-EP or SAQ D. Significantly increases security obligations. Rejected. |
