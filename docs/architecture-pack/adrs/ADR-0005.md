# ADR-0005: Authentication Pattern (Email/Password, JWT + Refresh Token, No OAuth v1)

**Status:** Accepted
**Date:** 2026-05-15
**Feature:** 002-revizr
**Human Approval Required:** YES — authentication architecture for paired
student–parent accounts with C7 child data requires human DPO sign-off
per constitution §2 and compliance condition C-001.
**Author:** solution-architect

---

## Context

Revizr has two authentication contexts (teacher/tutor access removed per spec sign-off 2026-05-15):
1. **Student accounts** — can be as young as age 9. **No student under the age of 17
   can create their own account.** Student accounts are always created by the parent.
   The parent is the account holder; the student is a dependent user.
2. **Parent accounts** — adult users who create and manage student accounts (up to 5
   sibling accounts per parent). The parent registers first; student accounts are
   created by the parent only.

**Spec sign-off decision (2026-05-15):** The parental consent mechanism is structural —
parents create all accounts for under-17s. This eliminates the two-phase pending_consent
flow for under-13s (which assumed students might attempt to self-register). The
simplified flow: parent registers → parent creates student sub-account → student
receives login credentials from parent. No student self-registration endpoint exists.

**Removed:** Teacher/tutor read-only access (F18) has been removed from the product
entirely per spec sign-off Q4 answer: "No teachers or tutors at all."

Key constraints:
- C6 data classification: passwords (hashed), session tokens, recovery codes must
  be stored in secrets manager only and never logged.
- No social login (Google, Apple) in v1. Social login deferred because: (a) OAuth
  with Google/Apple requires a privacy policy and ToS review that is a separate legal
  instrument; (b) under-13 users cannot have Google/Apple accounts without parental
  consent under those platforms' own ToS, creating a verification complexity; (c) the
  compliance team has not assessed social login providers as data processors for C7 data.
- The parental consent flow (F1-B, F1-C) requires a two-phase account creation:
  child account created in `pending_consent` state; no personal data collected or
  processed until the consent token is confirmed from the parent's email.
- JWT access tokens must be short-lived to limit the blast radius of token leakage;
  refresh tokens must be long-lived but revocable to avoid forcing re-authentication
  on every session.

---

## Decision

**Email/password authentication with short-lived JWT access tokens (15 minutes) and
long-lived rotating refresh tokens (30 days) stored in HTTP-only, Secure, SameSite=Lax
cookies.**

Implementation:
- **Password hashing:** Argon2id (winner of the Password Hashing Competition; OWASP
  recommended). Parameters: memory=65536, iterations=3, parallelism=4. Never bcrypt
  (weaker against GPU brute force).
- **JWT access token:** Signed with RS256 (asymmetric). Public key embedded in the
  Next.js frontend for local verification; private key stored in AWS Secrets Manager.
  Token payload: `{ sub: userId, role: 'student'|'parent', parentId?: string,
  linkedStudentIds?: string[], iat, exp }`. Access token lifetime: 15 minutes.
- **Refresh token:** Cryptographically random 32-byte token (UUID v4 equivalent).
  Stored in `refresh_tokens` table (hashed with SHA-256, not plaintext). HTTP-only
  cookie. Lifetime: 30 days. Token rotation: each `/auth/refresh` call invalidates
  the presented token and issues a new one (refresh token rotation). A presented
  refresh token that has already been rotated (replay attack) triggers immediate
  revocation of all refresh tokens for that user.
- **Email verification:** Required before account is active. 24-hour verification
  link. Under-13 accounts: email verification step is the parental consent email —
  the same token verifies the parent's email and confirms consent.
- **Parental consent token flow:**
  - Student registers → account created with `status: 'pending_consent'` → zero
    data collection until consent confirmed.
  - Consent email sent to parent's email address (provided during child registration).
  - Email contains a signed consent token (JWT, 24-hour TTL, `purpose: 'parental_consent'`).
  - Parent taps link → PO-01 flow → parent creates account → consent confirmed →
    student account transitions to `status: 'active'`.
  - If consent token expires: student registration data purged from database (no data
    retained without consent for under-13).
- **Rate limiting:** `/auth/login` and `/auth/refresh` endpoints rate-limited to 5
  attempts per IP per 15 minutes via `@fastify/rate-limit`. After 5 failures, CAPTCHA
  challenge required (hCaptcha accessibility mode, per WCAG 3.3.8 AA — no cognitive
  puzzle CAPTCHAs). Account lockout after 10 consecutive failures within 1 hour.
- **Password reset:** Time-limited reset token (1 hour, single-use) sent to
  registered email. Reset token is a cryptographically random 32-byte value stored
  as a SHA-256 hash in `password_reset_tokens` table with expiry.
- **Session management:** The `refresh_tokens` table records `user_id`, `token_hash`,
  `created_at`, `expires_at`, `last_used_at`, `ip_address_hash` (hashed, not raw IP
  per C3 classification), `user_agent_hash`. Users can see active sessions and revoke
  any session from account settings.

---

## Consequences

**Positive:**
- HTTP-only cookie storage for refresh tokens prevents JavaScript access —
  XSS attacks cannot steal the refresh token.
- Rotating refresh tokens detect replay attacks and automatically revoke all sessions
  on replay detection.
- Short-lived JWT access tokens (15 min) limit the window during which a stolen
  access token can be used — no database round-trip needed to validate most API
  requests.
- The two-phase account creation for under-13 users guarantees zero data collection
  without consent, satisfying UK GDPR Article 8 and the ICO's guidance.
- Argon2id is resistant to GPU-accelerated dictionary attacks — critical for a service
  storing passwords for users as young as age 9 (who may choose weak passwords).

**Negative / Trade-offs:**
- No social login means slightly higher registration friction compared to "Sign in with
  Google." Mitigated by the fact that: (a) the target parents (Priya, Siobhan) are
  motivated purchasers for whom a 30-second registration is acceptable; (b) social login
  introduces OAuth provider as a data processor not yet assessed for C7 child data.
- JWT RS256 requires key management. Private key in AWS Secrets Manager; rotation
  process must be documented. Mitigated by a defined key rotation procedure in the
  operate runbook.

---

## Alternatives Considered

| Alternative | Reason Rejected |
|---|---|
| **NextAuth.js (Auth.js)** | NextAuth handles OAuth providers and session management for Next.js. Rejected because: (a) NextAuth's session model is optimised for OAuth flows, not the custom two-phase parental consent flow; (b) Revizr's backend is a separate Fastify service, and Auth.js is designed to be colocated with Next.js API routes — routing authentication through the Next.js layer and then to Fastify adds unnecessary hops; (c) the parental consent token flow requires custom logic that NextAuth doesn't provide. |
| **Auth0 / Clerk / Supabase Auth** | Managed auth services reduce implementation effort. Rejected because: (a) these services process authentication data (email, device info, IP) outside the UK; DPA assessment for C7 child data is an additional compliance burden not budgeted; (b) these services are data processors under UK GDPR — adding them to the processing chain for under-13 users requires DPIA consideration; (c) the custom parental consent flow is difficult to implement within managed auth service constraints. |
| **HS256 JWT (symmetric signature)** | HS256 requires the same secret for signing and verification. The frontend verifying the JWT would need the shared secret — a security risk. RS256 allows the frontend to verify with the public key only; the private key never leaves the backend. Rejected on security grounds. |
| **Opaque session tokens (no JWT)** | Pure server-side session tokens require a database round-trip on every authenticated API request. With the performance target of p95 ≤ 800 ms and expected concurrent session load, adding a session lookup query to every request increases database pressure. JWT avoids this for stateless verification. Rejected on performance grounds. |
