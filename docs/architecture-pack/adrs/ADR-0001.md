# ADR-0001: Frontend Framework Selection (Next.js App Router)

**Status:** Accepted
**Date:** 2026-05-15
**Feature:** 002-revizr
**Author:** solution-architect

---

## Context

Revizr is a responsive progressive web application (PWA) serving two distinct
navigation contexts — student app and parent dashboard — from a single codebase.
The platform decision (Phase 5) has confirmed: web-only v1, PWA with service
worker, responsive at four breakpoints (320 px, 768 px, 1 024 px, 1 440 px),
Welsh (cy) secondary locale in v1, WCAG 2.2 AA, and hard performance budgets
(FCP ≤ 2 000 ms low-end, JS bundle ≤ 150 kB gzipped).

The question: which JavaScript meta-framework hosts the frontend?

Key constraints driving the decision:
1. Performance budget (150 kB initial JS gzipped) requires aggressive code splitting
   and server-side rendering of the initial HTML to meet FCP targets on low-end hardware.
2. Welsh (cy) locale must be a first-class routing concern in v1, not a runtime toggle.
3. PWA requirements (service worker, manifest) must be manageable within the framework.
4. The diagnostic async flow requires Server-Sent Events (SSE) or polling from the
   browser; the framework must support colocated API routes or an easy proxy pattern.
5. Static-asset self-hosting (platform constraint: no third-party CDN for any user data)
   is simplest when the frontend and API coexist in the same deploy unit.
6. The team is hiring TypeScript engineers; the framework must have first-class TypeScript
   support and a well-understood, hireable knowledge base.

---

## Decision

**Next.js 15 (App Router) with TypeScript.**

- The App Router (React Server Components) delivers SSR on every initial load,
  enabling sub-2 000 ms FCP on the low-end tier by sending pre-rendered HTML to
  the client — the client's CPU is never required to execute framework JS before
  the user sees content.
- Built-in `next/font` with `display: swap` satisfies the font strategy requirement
  (self-hosted, subset, no external CDN calls introducing GDPR risk).
- `next-intl` (or `next/i18n` with locale-based routing) handles the `en-GB`/`cy`
  routing requirement at the framework layer, with locale in the URL path
  (e.g., `/cy/home`), satisfying the WCAG 3.1.1 `<html lang>` requirement and
  enabling server-side locale resolution without client-side flash.
- Route handlers (`app/api/`) provide colocated API surface for BFF (Backend for
  Frontend) endpoints — SSE streaming for diagnostic polling is supported natively
  via Next.js `Response` streaming.
- `@next/bundle-analyzer` and `lighthouserc` integration give CI-enforced bundle
  size gating against the 150 kB budget.
- Service worker via `next-pwa` or a hand-authored worker with `next.config.js`
  `swcMinify` and static export for the offline shell.
- Deployment target: Node.js server (not static export) to support SSR and route
  handlers. Hosted on AWS Lambda / App Runner in eu-west-2 (London) or on a
  Vercel Pro plan configured for UK data residency (CDN edge restricted to UK PoPs
  per compliance C-012).

---

## Consequences

**Positive:**
- Server-side rendering eliminates the blank-page-on-load problem on low-end
  Chromebooks (Amara's device). HTML is meaningful on first byte.
- Code splitting is automatic at the route boundary; a student loading `/session`
  does not download the parent dashboard bundle.
- The App Router's `loading.tsx` and `Suspense` boundaries map directly to the
  design requirement for skeleton states within 200 ms of API call initiation.
- React Server Components allow the weakness map and session history to be rendered
  server-side, reducing client-side JS to interactive components only.
- Next.js is the most widely deployed React meta-framework; hiring and community
  support are strong.

**Negative / Trade-offs:**
- App Router introduces React Server Components, which require discipline to avoid
  "client boundary creep." Developers unfamiliar with the RSC model may default to
  `"use client"` on everything, erasing the performance benefit. Mitigated by a
  Build squad convention: every component is Server unless it needs interactivity,
  with `"use client"` requiring a comment justifying the choice.
- Next.js adds a framework layer above React; framework-specific bugs or version
  incompatibilities require framework team resolution. Mitigated by pinning to a
  minor version in `package.json` and upgrading on a defined review cadence.
- SSR increases server compute compared to a static SPA. Mitigated by aggressive
  use of `export const revalidate` for quasi-static pages (exam board selection
  data, subject lists) and React cache for per-request deduplication.

---

## Alternatives Considered

| Alternative | Reason Rejected |
|---|---|
| **Remix (Vite-based)** | Remix has strong SSR story and good DX, but smaller hiring pool than Next.js for UK startups. `next-intl` locale routing and `next/font` have no direct equivalents requiring custom implementation. Rejected in favour of Next.js ecosystem maturity. |
| **Vite + React SPA (no SSR)** | SPA mode fails the FCP budget on low-end hardware: 150 kB JS parses on a throttled ARM device in ~240 ms but hydration time on top of that exceeds the 2 000 ms FCP target. No SSR means the initial screen is blank until JS executes. Rejected. |
| **Astro** | Excellent for content-heavy static sites. Revizr's session and weakness map flows are highly interactive and stateful; Astro's island architecture requires significant manual orchestration for session state. Rejected as not the right fit for an application-heavy product. |
| **SvelteKit** | Technically capable, but Svelte is a smaller ecosystem with fewer production Next.js-equivalent hires in the UK market. Rejected on hiring risk grounds. |
