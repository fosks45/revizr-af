# ADR-0004: AI Provider Selection (Claude API via Server-Side Gateway)

**Status:** Accepted
**Date:** 2026-05-15
**Feature:** 002-revizr
**Human Approval Required:** YES — Anthropic Data Processing Agreement must be
confirmed in place before any real user data is sent to the Claude API. See
compliance condition C-013.
**Author:** solution-architect

---

## Context

Revizr's diagnostic engine is the core AI capability. Its job: parse an uploaded
school report (PDF/image/text, unstructured) or structured quiz results, and produce
a topic-level weakness map per subject and exam board. This is a document
comprehension and structured data extraction task, not a generative creative task.

The AI model must:
1. Accept long-form unstructured text (school report narratives, teacher comments)
   and produce a structured JSON output mapping weaknesses to a taxonomy of
   exam-board topics.
2. Operate on a system prompt that constrains the output schema — specifically, the
   output must NOT include any special-category data (health, SEN, disability
   indicators) from the source document. Only topic-weakness signals are permitted
   in the output (DPIA screening requirement BR-F2-02).
3. Be callable from a server-side gateway only — never directly from the browser
   (Principle 10: all model calls logged; compliance C-013: DPA required).
4. Produce results within the performance budget: p95 diagnostic job completion ≤ 60 s
   (low-end tier), ≤ 45 s (standard tier), with 90 s being the absolute spec.md NFR.
5. Support prompt caching for the repeated system prompt that includes the full topic
   taxonomy for each exam board and level. The topic taxonomy prompt is large (~8–12 k
   tokens per board/level combination) and is identical across all user invocations
   for the same board/level. Prompt caching is a cost governance requirement (§4 of
   the constitution: "Failing to cache a pattern that qualifies is a cost governance
   defect").

Additionally, the plain-English progress summary generation for the parent dashboard
(F10) requires a model invocation that takes structured data (topic scores, session
history) and produces a readable narrative. This is a lower-stakes, lower-latency
call.

---

## Decision

**Anthropic Claude API (claude-sonnet-4-5 for diagnostic; claude-haiku-3-5 for
progress summaries) via a server-side Fastify plugin acting as an instrumented gateway.**

Rationale:
1. **Document comprehension accuracy:** Claude's long-context window (up to 200k tokens)
   handles edge cases where a school report is verbose. Claude's instruction-following
   is strong for constrained JSON schema output with negative instructions ("do not
   include any health or SEN indicators in the output").
2. **Prompt caching:** Anthropic's prompt caching feature caches the static portion of
   the prompt (system prompt + topic taxonomy) across API calls. For a 10 000-token
   system prompt, prompt caching reduces cost by ~90% on the cached portion and latency
   by ~50%. This directly satisfies the cost governance requirement.
3. **Structured output:** Claude reliably produces valid JSON when the output schema
   is specified in the system prompt. The gateway validates the response against the
   expected Zod schema before persisting — if the response is malformed, the job is
   retried up to 2 times before escalating.
4. **Model tiering:** `claude-sonnet-4-5` for high-stakes document parsing (complex,
   requires reasoning across unstructured text); `claude-haiku-3-5` for progress
   summaries (simple structured-data-to-prose conversion, lower cost, lower latency).
   This tiering is a cost governance requirement: expensive models for high-stakes work,
   cheaper models for high-volume low-stakes work.
5. **UK data residency:** The Anthropic API is called from the Revizr backend in
   eu-west-2. The Data Processing Agreement with Anthropic must confirm: no training
   on customer data; no retention of input content beyond the current API call;
   adequate jurisdiction or SCCs for any non-UK processing. This is a pre-build
   blocker (Condition C-013). The gateway plugin must not be activated with real user
   data until the DPA is confirmed.

**Gateway implementation:**
- Every model call goes through `packages/ai-gateway/` in the monorepo.
- The gateway records to the `audit_log` table: `prompt_hash` (SHA-256 of the
  system+user prompt, not the raw prompt content), `response_hash`, `model_id`,
  `tokens_in`, `tokens_out`, `latency_ms`, `cost_usd`, `job_id`, `user_id_hash`.
  Raw prompts are never logged (compliance: C3+ data must not be in logs in plaintext).
- The gateway enforces `max_retries: 2` per call. On failure after 2 retries, the
  diagnostic job transitions to `failed` state and the user is offered the in-product
  diagnostic quiz (F3).
- Prompt caching is applied by structuring the system prompt as a cached "ephemeral"
  block and appending the user-specific document content as the non-cached user message.

---

## Consequences

**Positive:**
- Prompt caching dramatically reduces cost for the board/level system prompts that
  are identical across thousands of user invocations.
- Model tiering (Sonnet for diagnostic, Haiku for summaries) keeps the per-feature
  daily inference cost within the $200 cap at expected v1 volume.
- Claude's long context window means the full document can be passed in a single API
  call rather than requiring chunking logic.
- The gateway abstraction means the underlying model can be swapped (e.g., to a newer
  Claude model, or to an alternative provider) without changing any handler code.

**Negative / Trade-offs:**
- Dependency on a third-party AI provider introduces availability risk. Mitigated by:
  (a) the diagnostic job queue retries on API errors; (b) the fallback path to the
  in-product diagnostic quiz (F3) means the product is usable even if the AI API is
  unavailable.
- Anthropic's API is priced per token. At high volume (10 000+ diagnostic sessions/day),
  cost may approach the daily inference cap. Mitigated by prompt caching and the
  quarterly model review requirement in the constitution.
- The DPA with Anthropic is a pre-build blocker. If the DPA cannot be executed before
  the planned build start, the document upload path (F2) must be deferred and F3
  (in-product quiz) becomes the primary diagnostic path for the initial launch.

## Alternatives Considered

| Alternative | Reason Rejected |
|---|---|
| **OpenAI GPT-4o** | GPT-4o has comparable capability for document comprehension. Rejected in favour of Claude because: (a) Claude's instruction following for negative constraints ("do not include SEN data") is more reliable in internal evaluation; (b) Anthropic's prompt caching implementation is more explicit and auditable; (c) the agent-foundry platform has existing tooling for the Anthropic SDK (see CLAUDE.md). |
| **Google Gemini Pro** | Strong long-context model. Rejected because: (a) Google Cloud data residency for UK is less established than Anthropic's contractual controls; (b) the compliance team has not assessed Gemini's DPA terms for C7 child data; (c) adding a second cloud provider increases the compliance surface. |
| **Open-source model (Llama 3, Mistral) self-hosted** | Self-hosting eliminates third-party DPA concerns. Rejected because: (a) self-hosting a capable model on UK infrastructure (e.g., a GPU instance in eu-west-2) requires significant infrastructure investment not budgeted for v1; (b) document parsing accuracy of open-source 7B–70B models is materially lower than frontier models for this unstructured document comprehension task; (c) ongoing model maintenance (updates, security patches) is an operational burden. Scoped for v2 reassessment based on v1 cost data. |
| **Fine-tuned classifier (no frontier model)** | For the specific task of mapping school report text to a fixed topic taxonomy, a fine-tuned classifier could be built. Rejected because: (a) building training data for 6 exam boards × 4 levels × multiple subjects requires significant labelling investment; (b) the input domain (school reports) is highly varied and low-resource; (c) time-to-build is incompatible with v1 timeline. |
