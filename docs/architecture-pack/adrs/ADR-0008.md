# ADR-0008: File Storage and Deletion Policy (S3 UK + Immediate Post-Processing Delete)

**Status:** Accepted
**Date:** 2026-05-15
**Feature:** 002-revizr
**Author:** solution-architect

---

## Context

Revizr's core diagnostic feature (F2) requires users to upload school reports —
typically PDF or image files. These documents contain:

- Student name (C3)
- School name and teacher names (C3)
- Academic performance data (C1)
- Potentially medical or SEND information referenced in a report (C5 — Special
  Category under GDPR Article 9)

The data classification of an uploaded report is therefore **C5 at worst**, triggering
the requirements from the §8.5 taxonomy: sovereign data residency, field-level
encryption, explicit lawful basis, DPIA (Data Protection Impact Assessment), and
strict cross-border prohibition.

Compliance conditions from the Phase 4 compliance pack drive the storage decision:

- **C-004**: Raw user documents must not be retained beyond the minimum period
  necessary for processing. Post-processing, the original document is no longer
  required; only extracted weakness signals are retained in `diagnostic_results`.
- **C-008**: Documents containing C5 data (potential SEND or health references) must
  be deleted immediately upon processing completion. The system cannot distinguish
  a benign report from one containing C5 data at upload time; therefore all reports
  are treated as C5 and deleted immediately.

The question: which object store, and what is the exact deletion lifecycle?

---

## Decision

**AWS S3 (eu-west-2, London); presigned upload URLs from browser directly to S3;
BullMQ worker processes then immediately deletes the raw document. No raw report
content is retained after the processing job completes.**

Implementation specifics:

- **Storage location:** AWS S3 bucket in `eu-west-2` (London). Bucket policy
  explicitly denies replication to any other region. Server-side encryption:
  SSE-KMS with a customer-managed key (CMK) stored in AWS KMS eu-west-2.
- **Upload flow:** The Revizr API generates a presigned S3 PUT URL (15-minute
  expiry) and returns it to the browser along with a job ID. The browser uploads
  the file directly to S3 — the raw file never transits Revizr's API servers,
  minimising the attack surface and avoiding memory/disk buffering of C5 data on
  application infrastructure.
- **Processing:** The BullMQ worker picks up the job, fetches the object from S3
  using a server-side AWS SDK call (signed with an IAM role restricted to the
  specific bucket and key prefix), extracts text via a PDF parser, sends extracted
  text (not the original file) to the Claude API gateway for weakness analysis, and
  writes results to `diagnostic_results`.
- **Deletion:** Immediately after successful text extraction — before the Claude API
  call completes — the worker issues a `DeleteObject` call for the S3 key. The raw
  document is permanently deleted. The S3 key is recorded in `uploaded_documents`
  with `status = 'deleted'` and `deleted_at` timestamp. If the deletion call fails,
  the job retries the deletion (not the full pipeline) up to 3 times before
  escalating to the Operate squad alert channel.
- **S3 Lifecycle Rule (safety net):** An S3 lifecycle rule expires all objects in
  the bucket after 24 hours, regardless of application-level deletion. This is a
  belt-and-braces control — primary deletion is application-level, lifecycle rule
  is the fallback.
- **No retention:** `diagnostic_results` stores only derived weakness scores and
  evidence snippets extracted from the report. Evidence snippets are field-level
  encrypted (AES-256, per-record key). The original document is never stored beyond
  the processing window.
- **DPIA:** A Data Protection Impact Assessment is required before this feature is
  launched with real user data. The DPO must sign off on the lawful basis (legitimate
  interests — educational improvement) and the technical controls described here.

---

## Consequences

**Positive:**
- Immediate deletion satisfies C-004 and C-008 unconditionally: raw C5 data is
  never retained beyond the processing window.
- Presigned URL pattern eliminates raw file transit through Revizr API servers,
  reducing PII exposure surface and avoiding the need to handle large file uploads
  in the application tier.
- SSE-KMS with CMK in eu-west-2 satisfies the C5 encryption requirement (TLS in
  transit, at-rest encryption with customer-managed keys).
- S3 lifecycle rule as a safety net ensures deletion even if application code fails.
- The `uploaded_documents` audit trail (s3_key, content_hash, deleted_at) enables
  post-incident forensics without retaining content.

**Negative / Trade-offs:**
- Immediate deletion means the diagnostic result cannot be recalculated from the
  original document after the fact. If the Claude API produces a poor result on
  first pass, the document is gone. Mitigated by: (a) writing `content_hash` before
  deletion so users can re-upload the same document if needed, and (b) Claude API
  call occurring in the same job, so failure triggers retry before deletion.
- Workers must have IAM permissions to both read and delete S3 objects. These
  permissions must be scoped to the diagnostic bucket only (principle of least
  privilege). IAM policy reviewed as part of security hardening (Sprint 7).
- DPIA must be completed by a human DPO before v1 launch. This is a blocking
  human-approval item.

---

## Alternatives Considered

| Alternative | Reason Rejected |
|---|---|
| **Store files in the database (PostgreSQL bytea)** | Acceptable for small files but puts C5 binary data in the primary database alongside all other application data. Complicates backup retention policy (DB backups would retain documents beyond the processing window). Rejected. |
| **AWS S3 eu-west-1 (Ireland)** | Ireland is within the EU/EEA and would satisfy GDPR cross-border rules. However, compliance condition C-012 requires UK data residency (not just EU). EU-west-1 is excluded. |
| **Azure Blob Storage (UK South)** | Technically equivalent to S3 in eu-west-2 for residency purposes. Revizr's infrastructure is AWS-first; mixing cloud providers adds operational complexity for no material benefit in v1. Revisit if Azure compliance features become necessary. |
| **Retain documents for 30 days to allow re-processing** | Offers better recovery options but is incompatible with C-008 (immediate deletion of C5 data) and creates ongoing GDPR Article 5(e) storage limitation risk. Rejected. |
| **Process in-memory only, no S3** | Streaming the upload through the API, processing in-memory, and never writing to S3 would be cleanest. However, it tightly couples upload bandwidth to worker availability, creates memory pressure on API servers for large PDFs, and makes retry-on-failure impossible without asking the user to re-upload. Rejected. |
